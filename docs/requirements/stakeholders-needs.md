## <a name="article1">1. Вступ</a>

Цей документ надає описову інформацію для задоволення потреб зацікавлених в сервісі користувачів, в якості яких виступають аналітики та соціологи, проте, в проекті можуть бути зацікавлені студенти, викладачі, менеджери та інші, що ставлять собі за мету організацію опитування експертів (з подальшою роботою над ними).

### <a name="article11">1.1 Мета</a> 

Метою цієї сукупності вимог є формування функціонального та зручного сервісу, що дозволить його клієнтам з легкістю організовувати опитування експертів, та проводити обробку отриманих даних.  

### <a name="article12">1.2 Контекст</a>

Цей документ формує технічне завдання для проекту системи організації та управління опитуваннями експертів "SurvExp".

### <a name="article13">1.3 Основні визначення та скорочення</a>

## <a name="article2">2. Характеристика ділових процесів</a>

### <a name="article21">2.1 Призначення сервісу для опитувань</a>  

Функціональність сервісу опитувань призначена для надання послуг з організації повного циклу експертного опитування. Результатом роботи сервісу є отримання клієнтом зібраних даних.

### <a name="article22">2.2 Взаємодія з клієнтом</a>  

Клієнти можуть в будь-який момент зв’язатися з сервіс-менеджером, що обслуговує  користувачів через систему технічної підтримки для отримання необхідної допомоги у вирішенні виниклих питань.

### <a name="article23">2.3 Сценарії використання</a>

#### Сценарій №1  
**ID:** BC-1  
**НАЗВА:** Запит на створення нового опитування  
**УЧАСНИКИ:** Аналітик, Менеджер, Замовник  
**ПЕРЕДУМОВИ:** Немає  
**РЕЗУЛЬТАТ:** Опитування успішно додане  
**ВИКЛЮЧНІ СИТУАЦІЇ:**  
  -	EX-1: Аналітик не схвалив додання нового опитування  
  
**ОСНОВНИЙ СЦЕНАРІЙ:**  
1. Замовник передає менеджеру заявку на опитування.  
2. Менеджер реєструє замовлення.  
3. Менеджер повідомляє аналітику про нове замовлення.  
4. Аналітик аналізує доцільність реалізації замовлення (можливо EX-1).  
5. Аналітик повідомляє результат аналізу менеджеру.  
6. Менеджер повідомляє результат замовнику.  

#### Сценарій №2  
**ID:** BC-2  
**НАЗВА:** Редагування опитування  
**УЧАСНИКИ:** Аналітик, Менеджер, Замовник  
**ПЕРЕДУМОВИ:** Наявність опитування  
**РЕЗУЛЬТАТ:** Опитування успішно скориговане  
**ВИКЛЮЧНІ СИТУАЦІЇ:**  
  -	EX-2: Аналітик не схвалив зміни  
  
**ОСНОВНИЙ СЦЕНАРІЙ:**  
1. Замовник повідомляє менеджеру про бажання зробити зміни в опитуванні.  
2. Менеджер повідомляє аналітику про бажання замовника внести зміни.  
3. Аналітик переконується в коректності та доцільності змін (можливо EX-2).  
4. Аналітик повідомляє результат аналізу менеджеру.  
5. Менеджер підтверджує корегування, або, у випадку, EX-2 повідомляє причину скасування.  

#### Сценарій №3  
**ID:** BC-3  
**НАЗВА:** Перегляд статистики опитування  
**УЧАСНИКИ:** Замовник  
**ПЕРЕДУМОВИ:** Наявність результатів опитування  
**РЕЗУЛЬТАТ:** Отримання статистики  
**ВИКЛЮЧНІ СИТУАЦІЇ:** Відсутні  
**ОСНОВНИЙ СЦЕНАРІЙ:** Замовник переглядає статистику опитування  

#### Сценарій №4  
**ID:** BC-4  
**НАЗВА:** Надсилання запрошення для участі в опитуванні  
**УЧАСНИКИ:** Менеджер, Система  
**ПЕРЕДУМОВИ:** Наявність опитування  
**РЕЗУЛЬТАТ:** Експерт отримав персональне запрошення  
**ВИКЛЮЧНІ СИТУАЦІЇ:** Відсутні  
**ОСНОВНИЙ СЦЕНАРІЙ:**  
1. Менеджер обирає експертів, що отримають запрошення  
2. Система надсилає запрошення  

#### Сценарій №5  
**ID:** BC-5  
**НАЗВА:**  Отримання результату проходження опитування  
**УЧАСНИКИ:** Система, Експерт  
**ПЕРЕДУМОВИ:** Експерт заповнив анкету опитування  
**РЕЗУЛЬТАТ:** Система зберегла результат проходження опитування  
**ВИКЛЮЧНІ СИТУАЦІЇ:** Відсутні  
**ОСНОВНИЙ СЦЕНАРІЙ:**  
1. Експерт підтверджує відправлення анкети.  
2. Система отримує і зберігає результат опитування.  

#### Сценарій №6  
**ID:** BC-6  
**НАЗВА:** Нагадування про наближення закінчення терміну проведення опитування  
**УЧАСНИКИ:** Менеджер  
**ПЕРЕДУМОВИ:** Експерт не пройшов опитування. До закінчення терміну проведення опитування залишилась певна кількість часу (в залежності від опитування)  
**РЕЗУЛЬТАТ:** Експерт отримав повідомлення з нагадуванням  
**ВИКЛЮЧНІ СИТУАЦІЇ:** Відсутні  
**ОСНОВНИЙ СЦЕНАРІЙ:** Менеджер надсилає експерту повідомлення з нагадуванням про закінчення терміну проведення опитування  

## <a name="article3">3. Короткий огляд продукту</a>

**SurvExp** (від англ. *Survey of Experts* - опитування експертів) - це система організації та управління опитуваннями експертів, що полегшує взаємодію між зацікавленими особами та експертами. Доступ до платформи здійснюється через web-інтерфейс.  

## <a name="article4">4. Функціональність</a>

Основні вимоги до функціональності, що пред’являються зацікавленими особами до предмета розробки, відносяться до чотирьох категорій:  
- Користувач:  
  - Менеджер
  - Експерт  
  - Аналітик  
 
**4.1 Можливості «Користувач»:**  
- Реєструватися в системі;  
- Авторизуватися в системі.  

**4.1.1 Можливості «Менеджер»:** 
- Керувати процесом підготовки та проведення опитування. 
  - Надсилання запрошень для участі в опитуванні;
  - Надсилання персональних запрошень для участі в опитуванні;  
  - Створення запиту на створення опитування аналітиком.  

**4.1.2 Можливості «Експерт»:**  
- Проходження опитування.  

**4.1.3  Можливості «Аналітик»:**  
- Створювати опитування;  
- Додавати питання в опитування.  

## <a name="article5">5. Практичність</a>

Основні вимоги до практичності, що пред’являються зацікавленими особами до предмета розробки полягають в зручності інтерфейсу системи.  

## <a name="article6">6. Надійність</a>

Основні вимоги до надійності, що пред’являються зацікавленими особами до предмета розробки полягають в регулярному здійсненню резервного копіювання баз даних. 

## <a name="article7">7. Продуктивність</a>

Основні вимоги до продуктивності, що пред’являються зацікавленими особами до предмета розробки полягають в тому, щоб максимально можлива кількість експертів, що одночасно проходять опитування, доходила до 10 людей та система стабільно працювала.

## <a name="article8">8. Експлуатаційна придатність</a>

Основні вимоги до експлуатаційної придатності, що пред’являються зацікавленими особами до предмета розробки полягають в постійній підтримці системи, що полягає в тому, що протягом вказаного у договорі періоду часу систему можна буде модифікувати, відповідно до потреб зацікавлених осіб.
